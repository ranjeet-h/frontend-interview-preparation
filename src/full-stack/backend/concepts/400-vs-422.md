# 400 vs 422

## Detailed explanation

400 means the request is malformed or cannot be parsed; 422 means the request is syntactically valid but semantically invalid for the domain.

## 1. One-line mental model

400 is broken request shape; 422 is valid shape with invalid meaning.

## 2. Problem it solves

Validation errors are easier for frontends when parsing errors and business validation errors are separated.

## 3. Core idea

- Use 400 for invalid JSON, missing content type, malformed query params.
- Use 422 for field validation failures after parsing succeeds.
- 422 often returns field-level error details.
- Some teams use 400 for all validation; consistency matters most.
- Document the project convention.

## 4. Visual / analogy

```txt
400 = unreadable form. 422 = readable form with wrong values.
```

## 5. Minimal example

```txt
return res.status(422).json({ errors: { email: "Invalid email" } });
```

## 6. Real-world example

Registration payload with malformed JSON returns 400; valid JSON with weak password returns 422.
