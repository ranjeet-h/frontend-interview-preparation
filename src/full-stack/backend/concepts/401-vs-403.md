# 401 vs 403

## Detailed explanation

401 means the request is unauthenticated or has invalid credentials; 403 means the user is authenticated but not allowed to perform the action.

## 1. One-line mental model

401 asks who you are; 403 says you are known but not allowed.

## 2. Problem it solves

Auth bugs become confusing when APIs return 403 for missing login or 401 for permission failure.

## 3. Core idea

- Return 401 when token/cookie is missing, expired, or invalid.
- Return 403 when identity is valid but role/permission is insufficient.
- 401 can trigger login or refresh-token flow.
- 403 should show access denied, not silently refresh forever.
- Never expose sensitive permission details in the error body.

## 4. Visual / analogy

```txt
401 = no valid badge. 403 = valid badge, wrong room.
```

## 5. Minimal example

```txt
if (!user) return res.sendStatus(401); if (!user.isAdmin) return res.sendStatus(403);
```

## 6. Real-world example

Admin endpoint: no token returns 401; logged-in normal user returns 403.
